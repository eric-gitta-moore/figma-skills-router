# Skill Router Creator

使用这个 skill 来创建“单入口 router skill”，把一组已有 skills 聚合成一个统一、稳定、可维护的对外入口。

适用场景包括：

- 你手上已经有多个 skills，但调用方不知道该先选哪个
- 多个 skills 来自上游仓库，需要保留同步能力
- 某些 skills 之间存在前置关系、组合关系或互相跳转
- 你希望对外只暴露一个入口 skill，把复杂性隐藏在内部结构里

## 目标

产出一个可长期维护的 skill router，而不是临时拼接的文档集合。

最终结果通常包含：

- 一个对外暴露的 router skill
- 一组作为参考资料的下游 skills 镜像
- 明确的分流规则
- 明确的组合依赖关系
- 明确的上游同步方式

## 核心方法论

### 1. 只暴露一个统一入口

对外只保留一个真正会被识别的 skill 入口。

这个入口负责：

- 识别用户真正想要的最终产出物
- 将请求路由到最合适的 guide 或 workflow
- 决定是否需要组合加载多个 guide

不要让调用方直接面对一长串平级 skills，否则 router 的价值就不存在了。

### 2. 按“最终交付物”分流，而不是按工具名分流

router 不应该先问“要用哪个工具”，而应该先判断“用户最后想得到什么”。

优先按以下维度拆分：

- 最终产物是代码
- 最终产物是某个平台内的实体对象
- 最终产物是整页 / 整屏 / 多步骤工作流
- 最终产物是配置、规则、模板或映射

只有当交付物相同、但方法不同的时候，才再细分到具体工具层。

### 3. 把下游 skills 视为参考实现，而不是对外入口

如果需要聚合的是一批已有 skills，推荐把它们放到 router skill 的 `references/` 下，作为参考资料和工作流来源。

这样做的好处：

- 对外入口稳定
- 内部结构清晰
- 上游内容可以整体镜像
- router 可以只引用必要内容，而不必重写全部正文

### 4. 下游资料尽量原样保留，只做机械化包装

对下游 skill 的整理要尽量克制。

优先只做这些事情：

- 调整入口文件命名，避免被当成独立 skill 重复识别
- 修正文档中的相对链接
- 修复已知错链、错名或目录别名
- 保留原始 `references/`、`scripts/`、资源文件和示例

不要一上来就重写下游正文。重写越多，后续越难同步上游。

### 5. 明确区分“对外文案”和“仓库内部实现细节”

router 的 `SKILL.md` 是给调用方看的，不是给仓库维护者看的。

因此：

- 对外 skill 文案要讲“什么时候用我、我怎么分流、我会加载什么”
- 不要在对外入口里解释内部目录重命名、包装策略、识别规避等实现细节

内部实现细节应该写在项目 README、维护文档或同步脚本说明里。

### 6. 组合依赖必须显式声明

如果某个高层 workflow 依赖一个基础 skill，不要隐含约定，必须在 router 里直接写清楚。

典型形式：

- `A` 场景必须组合 `B`
- `C` workflow 依赖 `D` 作为底层执行能力
- `E` guide 是主流程，`F` guide 是底层约束

这样可以减少误用，也方便后续替换底层实现。

## 标准工作流

按下面的顺序创建 skill router，不要跳步。

### Step 1：盘点候选 skills

先收集所有需要被聚合的 skills，并回答这些问题：

- 每个 skill 的主要交付物是什么
- 哪些 skill 是顶层工作流
- 哪些 skill 是底层能力
- 哪些 skill 只适合作为参考资料
- 哪些 skill 存在组合依赖
- 哪些 skill 的边界有重叠

如果连这些边界都没理清，就不要急着写 router。

### Step 2：抽象路由维度

不要直接按已有 skill 名称罗列目录，要先提炼一套稳定的路由维度。

推荐优先使用：

- 按最终交付物
- 按任务阶段
- 按抽象层级
- 按单一任务与复合任务

最终 router 的章节结构应该由这些维度决定，而不是由上游目录名决定。

### Step 3：确定公共入口和 references 结构

推荐结构：

```text
skills/
  <router-skill-name>/
    SKILL.md
    references/
      <skill-a>/
      <skill-b>/
      <skill-c>/
```

约束：

- `SKILL.md` 只保留一份，作为唯一入口
- `references/` 下放镜像资料
- 下游资料入口使用不会被识别成独立 skill 的命名

### Step 4：镜像下游 skills

将下游 skills 复制或同步到 `references/` 中。

推荐保留：

- 主文档
- `references/`
- `scripts/`
- 类型文件
- 示例资源

尽量不要删除原始辅助资料，否则后续 router 会失去上下文。

### Step 5：做最小包装

对镜像下游 skill 做最少量整理：

- 将下游入口文档改成非技能入口命名，例如 `GUIDE.md`
- 批量修正文档内的相对链接
- 修复目录名、别名和已知错链

这一层的目标是“可引用、可导航、可同步”，不是“重写一套新的技能体系”。

### Step 6：编写 router 的 `SKILL.md`

router 入口文档至少要包含以下内容：

- 这个 router skill 的用途
- 如何按任务目标分流
- 每个分支要加载哪个 guide
- 哪些场景需要组合加载
- 冲突场景下的优先级
- 加载纪律，例如“只加载必要 guide，不要默认全量加载”

router 文档应该短、清晰、面向使用者。

### Step 7：补项目级 README

如果这个 router 是一个独立项目，建议在项目根目录补一份 README，讲清楚：

- 项目定位
- 目录结构
- router 与 references 的关系
- 上游来源
- 同步策略

这些内容不应该堆进对外 skill 文档里。

### Step 8：验收

交付前至少检查以下项目：

- 是否只有一个对外可识别的 skill 入口
- 下游 references 是否完整
- references 内是否还残留会被错误识别的入口文件
- router 到各 guide 的链接是否有效
- 组合依赖是否写清楚
- 文档是否把内部实现细节泄露给调用方
- 后续是否可以低成本同步上游

## 常用分流模板

下面这套模板适合大多数 skill router：

### 模板一：按最终产出物

- 产出是代码
- 产出是平台内对象
- 产出是完整页面或工作流
- 产出是规则、模板或配置

### 模板二：按任务层级

- 顶层业务工作流
- 通用执行能力
- 辅助资料与参考实现

### 模板三：按是否需要组合加载

- 单 guide 即可完成
- 主 guide + 底层 guide
- 多 guide 串联

## 不要这样做

- 不要对外暴露一堆平级 skill，让用户自己猜
- 不要在 router 的 `SKILL.md` 里堆内部维护说明
- 不要为了包装而重写全部下游正文
- 不要忽略 skill 之间的组合依赖
- 不要把“目录结构”误当成“路由策略”
- 不要让上游一更新就需要人工重新手改大量文档

## 输出要求

当你用这个 skill 创建一个新的 router skill 时，最终至少要产出：

1. 一个新的 router skill 目录
2. 一份面向调用方的 `SKILL.md`
3. 一套整理好的 `references/`
4. 一份项目 README 或维护说明
5. 一套清晰的同步和验收规则
